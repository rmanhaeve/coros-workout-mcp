# KEY FINDING: POST /training/plan/add only creates a Plans-library TEMPLATE
# (inSchedule 0, startDay null). To put workouts on the CALENDAR (and thus sync
# them to the watch) you must "execute" that template onto a start date. The
# full lifecycle is four undocumented endpoints:
#
#   1. POST /training/plan/add                      -> build day-offset template
#      (body fully decoded in research/schedule-plan-request-all.txt)
#      returns: {"result":"0000","data":"<templatePlanId>"}
#
#   2. POST /training/schedule/executeSubPlan?startDay=YYYYMMDD&subPlanId=<templatePlanId>
#      body: {}   (empty)  -> copies the template onto the calendar; each plan
#      workout lands on startDay + its dayNo. Creates an independent calendar
#      "sub-plan instance" (new plan/program ids). returns {"result":"0000"}.
#
#   3. POST /training/plan/delete
#      body: ["<planId>", ...]   -> deletes plan TEMPLATES only. Verified: this
#      does NOT remove already-executed calendar entries.
#
#   4. POST /training/schedule/quitSubPlan?subPlanId=<calendarSubPlanInstanceId>
#      body: {}   -> "exit plan": removes an executed sub-plan from the calendar.
#      The id is the scheduled entity's planId (from schedule/query), NOT the
#      template id. Distinct from plan/delete.
#
# Reads:
#   GET /training/plan/detail?id=<id>&region=3&supportRestExercise=1  (template)
#   GET /training/schedule/query?startDate=YYYYMMDD&endDate=YYYYMMDD&supportRestExercise=1
#       -> data.entities[] are the scheduled workouts; each entity.planId is the
#          sub-plan instance id to pass to quitSubPlan. (In the calendar, dayNo
#          is an ABSOLUTE day index, not the 0-based offset used in a template.)
#   POST /training/plan/query   body {"name":"","startNo":0,"limitSize":N}
#       -> lists library templates AND executed instances.
#
# The MCP `schedule_workouts` tool runs 1 -> 2 -> 3: build template, execute at
# startDay, delete the redundant template (calendar copy survives). Entity
# happenDay is left "" — placement is by startDay + dayNo; a happenDay that
# disagrees with startDay+dayNo misplaces the workout (observed).

# Same body as /training/program/add but with the existing program `id` set.

# --- POST /training/plan/update (edit a plan in place) ---
# A full-document PUT: echo the plan/detail document back with modifications.
# Key fields: top-level `id` (plan id), the full entities[]/programs[], and a
# DIFFERENTIAL `versionObjects[]` describing what changed:
#   status 1 = newly added entity   (also carries onlyId uuid + new program)
#   status 2 = existing/modified entity (moved day, etc.)
#   (unchanged entities are omitted from versionObjects)
# Recompute totalDay (= max dayNo + 1) and min/maxWeeks (min/max workouts per
# content-bearing week). Verified: moving a workout to another day with a single
# {id, planProgramId, planId, status:2, type:0} versionObject -> {"result":"0000"}
# and plan/detail reflects the new dayNo.
#   body: { ...plan/detail..., "id":"<planId>", entities:[...moved...],
#           versionObjects:[{id,planProgramId,planId,status:2,type:0}], ... }
